## 一个常见误解

如果你写过最简单的 LLM 调用，大概是这样：

```python
from langchain.chat_models import init_chat_model

model = init_chat_model("openai:gpt-5.5", temperature=0)
response = model.invoke("北京今天天气怎么样？")
print(response.content)
```

跑起来，模型会"回答"——但它其实**不知道**北京今天的天气。它只是在概率上生成了一段看起来像答案的文字。这就是裸 LLM 调用的第一个局限：**没有外部能力**。

于是你听过一句话：**"Agent 就是把 LLM 放进一个 `while` 循环里，让它反复调用工具直到任务完成。"**

这句话对，但只对了一半。它描述了 Agent 的**形**，没讲清 Agent 的**神**。要理解差在哪，先把"形"画出来。

上面的交互式 Demo（目前为骨架占位，正式 trace 待后端运行时接入）对比了两种执行轨迹：

| 维度 | 裸 LLM 调用 | 带工具循环的 Agent |
|---|---|---|
| 调用次数 | 一次 | 多次（循环） |
| 外部能力 | 无 | 可调工具（搜索、计算、读写文件……） |
| 状态 | 无记忆，每次独立 | 跨轮保留消息与中间结果 |
| 终止 | 调完即停 | 模型自行决定"做完了" |
| 失败 | 直接报错 | 需要恢复策略（重试/回退） |

可以看到，"while 循环"只是这张表里**调用次数**那一行。其余四行——外部能力、状态、终止、失败恢复——才是 Agent 真正比裸调用多的东西，而它们恰恰是循环本身解决不了的。

## 把 Agent 拆成四块

抛开框架，一个 Agent 在概念上由四部分组成：

1. **LLM**——大脑，负责推理与决策"下一步做什么"。
2. **工具（Tools）**——手脚，让模型能对外部世界产生影响、获取信息。
3. **循环（Loop）**——心跳，反复"思考→行动→观察"直到任务完成。
4. **状态（State）**——记忆，跨轮保留对话与中间结果。

这就是经典的 **ReAct 模式**：**Reason**（思考）→ **Act**（行动）→ **Observe**（观察），循环往复。

用伪代码表示：

```text
while True:
    thought = LLM(messages)              # 思考：下一步该干什么
    if thought 不请求任何工具:
        return thought                    # 做完了，输出最终答案
    tool_call = thought.tool_calls[0]
    result = 执行工具(tool_call)          # 行动：调用工具
    messages.append(result)               # 观察：把结果塞回上下文
```

看起来确实"只是个 while 循环"。那为什么市面上的 Agent 框架（LangGraph、Deep Agents）要写几万行代码？因为**真正生产可用的 Agent，还要解决循环本身管不了的三件事**。

## 循环管不了的三件事

### 1. 上下文工程

每跑一轮循环，消息列表就长一截。工具返回的内容、模型的思考、历史对话，全往上下文里塞。跑个十几轮，上下文就爆了——要么超 token 上限直接报错，要么成本飙升，要么早期关键信息被挤掉导致模型"失忆"。

**上下文工程**（Context Engineering）就是研究：什么该进上下文、什么该被压缩、什么该被外置到文件/数据库按需读取。这不是"循环"能解决的，它是一个独立的工程问题。

> 你可以把上下文想象成 Agent 的**工作台**。裸循环是"把所有东西都摊在台子上"，台子塞满了就崩。上下文工程是"学会把资料分类放进抽屉，需要时再取"——这正是第 6 章 Deep Agents 虚拟文件系统要解决的核心问题。

### 2. 状态与恢复

裸循环跑在内存里，进程一崩，全部状态丢失。用户聊到一半的网络中断了、服务器重启了、模型某一步抛错了——对话就断了，没法续上。

生产级 Agent 需要**检查点**（checkpoint）：把状态持久化下来，能中断、能恢复、能"时间旅行"回退到某一步重跑。这是循环结构里没有的一层。

### 3. 失败恢复

模型不总是听话。它会：

- 幻觉调用一个**不存在的工具**
- 传**错误参数**让工具抛异常
- 陷入**无限循环**反复调同一个工具
- 在不该停的时候**提前停**

裸循环遇到这些，要么崩，要么死循环。框架要做的，是给每一步加护栏：工具不存在时的兜底、参数校验、循环次数上限、异常重试。这些也不是"循环"本身能提供的。

## 用三个层次理解现有方案

为了让你在后续章节不至于迷失，我们把市面上的方案分成三个层次——这也是理解 LangChain / LangGraph / Deep Agents 关系的关键：

| 层次 | 角色 | 代表 | 你要写什么 |
|---|---|---|---|
| **Runtime（运行时）** | 提供状态机、检查点、流式等基础设施 | LangGraph | 节点、边、状态定义 |
| **Framework（框架）** | 在 Runtime 上封装常见 Agent 范式 | LangChain 的 `create_agent` | 模型 + 工具，范式是定好的 |
| **Harness（套件）** | 把生产级能力（上下文管理、子 Agent、记忆）打包成开箱即用 | Deep Agents | 几乎只写业务工具和提示词 |

打个比方：

- **Runtime** 是发动机和底盘——你得自己拼管线、挂挡（写图）。
- **Framework** 是整车——发动机和变速箱装好了，你只要踩油门（`create_agent`）。
- **Harness** 是带自动驾驶的整车——连导航、变道、避障都帮你做了，你只说目的地（`create_deep_agent`）。

理解了这个分层，你就明白本博客的章节顺序为什么这么排了。

## 小结

1. **Agent ≠ LLM + while 循环**。循环只是 Agent 的"形"，真正让它成立的是工具、状态、终止与失败恢复的工程化处理。
2. **ReAct 模式**（思考→行动→观察）是 Agent 的运行骨架，但裸循环走不远。
3. **循环管不了的三件事**：上下文工程、状态恢复、失败恢复——这正是框架存在的理由。
4. **三个层次**：Runtime（LangGraph）→ Framework（LangChain `create_agent`）→ Harness（Deep Agents），后续章节按此递进。
